# Multi-Tenant Notes App - 요구사항 정의서 v2

## 1. 프로젝트 개요

멀티 테넌트 아키텍처를 학습하기 위한 SaaS 메모/노트 애플리케이션.
테넌트(조직) 단위로 데이터를 격리하고, 플랜별 기능 제한과 역할 기반 권한을 구현한다.

## 2. 기술 스택

### Backend
- Spring Boot 3.x
- Spring Data JPA + QueryDSL
- PostgreSQL (RLS 적용)
- Flyway (DB 마이그레이션)
- JWT 인증 (Access + Refresh Token)
- Hibernate Envers (감사 추적)
- 멀티 모듈 구조 (common, storage)

### Frontend
- Next.js (Pages Router)
- TypeScript
- Tailwind CSS + shadcn/ui
- Zustand (상태 관리)
- Axios
- yarn (패키지 매니저)
- next-intl (i18n)

### Infrastructure
- Docker Compose (PostgreSQL)
- 서브도메인 기반 테넌트 식별

## 3. 사용자 역할

### 3.1. 슈퍼어드민 (SUPER_ADMIN)
- 시스템 전체를 관리하는 최상위 관리자
- 테넌트 외부에 존재 (특정 테넌트에 소속되지 않음)
- 별도의 어드민 대시보드 사용

### 3.2. 테넌트 내 역할

| 역할 | 설명 |
|------|------|
| OWNER | 테넌트 소유자. 테넌트 생성 시 지정됨. 테넌트당 1명 |
| ADMIN | 테넌트 관리자. OWNER가 지정. 복수 가능 |
| MEMBER | 일반 멤버. 초대를 통해 가입 |

## 4. 기능 요구사항

### 4.1. 슈퍼어드민 기능

| 기능 | 설명 |
|------|------|
| 테넌트 생성 | 테넌트 이름, 서브도메인, 플랜, OWNER 이메일 지정 |
| 테넌트 목록/상세 조회 | 전체 테넌트 현황 (멤버 수, 노트 수, 플랜, 상태) |
| 테넌트 설정 변경 | 플랜 변경, 활성/비활성 |
| 멤버 초대 링크 발급 | 특정 테넌트에 대한 초대 링크 생성 |
| 대시보드 | 전체 테넌트 통계 (총 테넌트 수, 총 사용자 수, 플랜별 분포) |

### 4.2. 인증/계정

| 기능 | 설명 |
|------|------|
| 초대 링크를 통한 회원가입 | 초대 링크 클릭 → 이메일/비밀번호 입력 → 해당 테넌트에 자동 소속 |
| 로그인 | 이메일 + 비밀번호 → JWT 발급 (Access + Refresh) |
| 토큰 갱신 | Access Token 만료 시 Refresh Token으로 자동 갱신 |
| 로그아웃 | Refresh Token 무효화 |
| 내 정보 조회/수정 | 이름, 비밀번호 변경 |

### 4.3. 멤버 관리 (OWNER/ADMIN 권한)

| 기능 | 설명 |
|------|------|
| 멤버 초대 링크 발급 | OWNER/ADMIN이 테넌트 내 초대 링크 생성 |
| 멤버 목록 조회 | 테넌트 내 전체 멤버 목록 (역할, 가입일) |
| 멤버 역할 변경 | OWNER만 가능. MEMBER ↔ ADMIN 변경 |
| 멤버 비활성화 | OWNER/ADMIN이 멤버 비활성화 (삭제 대신) |

### 4.4. 노트 CRUD

| 기능 | 설명 |
|------|------|
| 노트 작성 | 제목, 본문, 태그 지정 |
| 노트 목록 조회 | 본인 노트 + 공유받은 노트. 검색/태그 필터/페이징 |
| 노트 상세 조회 | 본문, 태그, 댓글, 공유 상태 |
| 노트 수정 | 본인 노트만 (OWNER/ADMIN은 모든 노트) |
| 노트 삭제 | 본인 노트만 (OWNER/ADMIN은 모든 노트). Soft delete |

### 4.5. 태그

| 기능 | 설명 |
|------|------|
| 태그 생성 | 노트 작성/수정 시 태그 지정 (없으면 자동 생성) |
| 태그 목록 조회 | 테넌트 내 전체 태그 목록 |
| 태그별 노트 필터링 | 특정 태그가 달린 노트 목록 |

### 4.6. 노트 공유

| 기능 | 설명 |
|------|------|
| 테넌트 내 공유 | 특정 멤버에게 노트 공유 (읽기 전용) |
| 외부 공유 링크 | 공개 URL 생성 (Pro 플랜만). 비회원도 열람 가능 |
| 공유 해제 | 멤버 공유 해제, 외부 링크 비활성화 |

### 4.7. 댓글

| 기능 | 설명 |
|------|------|
| 댓글 작성 | 노트에 댓글 추가 (본인 노트 또는 공유받은 노트) |
| 댓글 목록 조회 | 노트별 댓글 목록 (작성자, 작성일) |
| 댓글 수정/삭제 | 본인 댓글만 (OWNER/ADMIN은 모든 댓글 삭제 가능) |

### 4.8. 알림

| 기능 | 설명 |
|------|------|
| 알림 생성 | 이벤트 발생 시 자동 생성 (아래 알림 트리거 참고) |
| 알림 목록 조회 | 본인에게 온 알림 목록 (읽음/안읽음 필터) |
| 읽지 않은 알림 수 | 사이드바 뱃지에 표시할 미읽 카운트 |
| 알림 읽음 처리 | 개별 읽음, 전체 읽음 |
| 알림 삭제 | 개별 삭제 |

알림 트리거:

| 이벤트 | 수신자 | 메시지 예시 |
|--------|--------|------------|
| 노트 공유 받음 | 공유 대상 멤버 | "홍길동님이 '회의록' 노트를 공유했습니다" |
| 내 노트에 댓글 | 노트 작성자 | "김철수님이 '회의록'에 댓글을 남겼습니다" |
| 공유받은 노트에 댓글 | 공유 받은 멤버들 | "이영희님이 '회의록'에 댓글을 남겼습니다" |
| 멤버 역할 변경 | 대상 멤버 | "역할이 MEMBER에서 ADMIN으로 변경되었습니다" |
| 새 멤버 참가 | OWNER, ADMIN | "박민수님이 테넌트에 참가했습니다" |

### 4.9. 테넌트 설정 (OWNER 권한)

| 기능 | 설명 |
|------|------|
| 테넌트 정보 조회 | 테넌트 이름, 플랜, 사용량 현황 |

## 7. 멀티 테넌트 핵심 요구사항

### 7.1. 데이터 격리
- 공유 DB, 공유 스키마, Row-level 격리 (tenant_id 기반)
- PostgreSQL RLS를 DB 레벨 안전망으로 적용
- 모든 비즈니스 엔티티에 tenant_id 필수

### 7.2. 테넌트 식별
- 서브도메인 기반: tenant1.localhost:3000
- 백엔드 요청 시 X-Tenant-Id 헤더 전달
- TenantInterceptor → TenantContext (ThreadLocal)에 저장

### 7.3. 보안
- JWT에 tenantId 포함 → 요청 헤더의 tenantId와 일치 여부 검증
- 사용자가 소속된 테넌트 이외의 데이터 접근 차단
- Repository 레이어에서 tenant_id 자동 필터링

### 7.4. 성능 격리
- 테넌트 단위 API rate limiting
- 플랜별 사용량 체크 (노트 수, 멤버 수)
